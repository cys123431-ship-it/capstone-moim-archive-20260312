# 웹 구현 설명서

이 문서는 우리 팀의 웹 작업을 아주 쉽게 설명하기 위한 기록입니다.

중요한 원칙은 2가지입니다.

1. 처음에는 어렵게 시작하지 않고 `HTML + CSS + JavaScript`만으로 화면 뼈대를 만든다.
2. 나중에 웹 기능이 추가될 때마다 이 문서에 `무엇을`, `왜`, `어떻게` 만들었는지 계속 이어서 적는다.

앞으로 웹 관련 작업이 추가되면 이 문서 뒤에 계속 내용을 붙이면 됩니다.

---

## 1. 지금 만든 웹의 목표

이번 1단계는 `완성 서비스`를 만드는 것이 아니라, 아래 4가지를 먼저 눈에 보이게 만드는 단계입니다.

1. 모임 웹사이트가 어떤 화면으로 이루어지는지 보여주기
2. 모임 목록이 카드 형태로 보이게 하기
3. 검색 조건에 따라 목록이 바뀌는 흐름 익히기
4. 나중에 DB, 지도 API, 결제 API를 붙일 자리를 미리 만들어 두기

즉, 지금은 `기능 연습이 가능한 뼈대`라고 보면 됩니다.

---

## 2. 파일 구조

현재 파일은 이렇게 나누었습니다.

```text
Teamwork/
├─ index.html
├─ styles.css
├─ script.js
└─ docs/
   └─ web-implementation-guide.md
```

각 파일 역할은 아주 간단합니다.

- `index.html`: 화면의 뼈대
- `styles.css`: 화면 꾸미기
- `script.js`: 버튼 클릭, 검색, 카드 출력 같은 동작 처리
- `docs/web-implementation-guide.md`: 구현 설명 기록

---

## 3. 왜 이렇게 3개 파일로 나눴는가

웹을 처음 만들 때 한 파일에 다 넣을 수도 있습니다.
하지만 배우기에는 오히려 나누는 편이 더 좋습니다.

- HTML은 `화면 구조`
- CSS는 `디자인`
- JavaScript는 `동작`

이렇게 역할이 나뉘면, 나중에 문제가 생겼을 때 어디를 봐야 하는지 바로 알 수 있습니다.

예를 들어:

- 글자가 안 보이면 HTML 또는 CSS를 본다
- 버튼을 눌렀는데 반응이 없으면 JavaScript를 본다
- 배치가 이상하면 CSS를 본다

## 4. index.html 설명

`index.html`은 브라우저에 보이는 큰 구조를 만든 파일입니다.

지금은 아래 순서로 구성했습니다.

1. 상단 메뉴
2. 첫 화면 소개 영역
3. 현재 상태 카드
4. 모임 검색 영역
5. 지도 API가 들어갈 자리
6. 내 모임 영역
7. 모임 만들기 폼
8. 하단 푸터

순서의 이유는 다음과 같습니다.

- 사용자는 먼저 이 서비스가 무엇인지 봅니다.
- 그 다음 모임을 찾습니다.
- 그 다음 내 모임을 봅니다.
- 마지막으로 새 모임을 만듭니다.

즉, 사용자의 흐름대로 위에서 아래로 내려가게 구성한 것입니다.

### 4-1. 상단 메뉴

상단 메뉴는 이 부분입니다.

```html
<nav class="menu">
  <a href="#search">모임 찾기</a>
  <a href="#map">지도 자리</a>
  <a href="#mine">내 모임</a>
  <a href="#create">모임 만들기</a>
</nav>
```

여기서 중요한 점은 `href="#search"` 같은 형태입니다.

이 뜻은:

- 다른 페이지로 가는 것이 아니라
- 현재 페이지 안의 특정 위치로 이동한다는 뜻입니다.

즉, 메뉴를 누르면 해당 섹션으로 바로 내려갑니다.

### 4-2. 소개 영역

첫 화면에서는 이 웹사이트가 무엇인지 짧게 설명합니다.

이 부분이 필요한 이유는 사용자가 처음 들어왔을 때 아래 내용을 바로 알아야 하기 때문입니다.

- 이 사이트가 무슨 사이트인지
- 지금 어디까지 구현된 상태인지
- 무엇을 할 수 있는지

### 4-3. 검색 영역

검색 영역에는 아래 입력칸을 넣었습니다.

- 키워드
- 카테고리
- 지역

이 구조를 미리 만들어 두면, 나중에 DB가 생겼을 때 입력값만 서버로 보내면 됩니다.
즉, 지금의 검색창은 나중에 실제 검색 기능으로 바뀌기 쉬운 형태입니다.

### 4-4. 지도 자리

아직 지도 API는 붙이지 않았습니다.
하지만 미리 자리를 만들어 둔 이유는 화면 구조를 먼저 잡아야 나중에 API를 붙일 때 덜 헷갈리기 때문입니다.

지금은 `MAP PLACEHOLDER`라는 글자만 보이지만,
나중에는 이 자리에 지도 SDK가 들어옵니다.

### 4-5. 내 모임 영역

모든 모임과 별개로 `내가 참여한 모임`만 따로 보여주는 영역입니다.

이걸 따로 분리한 이유는:

- 사용자가 전체 목록에서 계속 찾지 않아도 되고
- 마이페이지 느낌의 기본 구조를 미리 만들 수 있기 때문입니다.

### 4-6. 모임 만들기 폼

여기서는 사용자가 직접 입력하는 흐름을 연습하기 위해 폼을 만들었습니다.

입력 항목은 아래와 같습니다.

- 모임 이름
- 카테고리
- 지역
- 날짜
- 모집 인원

이 폼은 나중에 백엔드로 연결되면 실제 등록 기능의 시작점이 됩니다.

## 5. styles.css 설명

`styles.css`는 화면을 꾸미는 파일입니다.

초보자가 가장 먼저 이해해야 할 것은 `class 이름을 기준으로 모양을 바꾼다`는 점입니다.

예를 들어:

```css
.meeting-card {
  display: grid;
  gap: 14px;
  padding: 22px;
}
```

이 코드는 `meeting-card`라는 이름을 가진 HTML 요소에게 아래 모양을 주는 뜻입니다.

- `display: grid;`
  화면 안쪽 요소를 격자처럼 정리
- `gap: 14px;`
  안쪽 요소끼리 간격 주기
- `padding: 22px;`
  카드 안쪽 여백 만들기

### 5-1. 왜 CSS 변수를 썼는가

맨 위에 이런 코드가 있습니다.

```css
:root {
  --bg: #f5f1e8;
  --accent: #14532d;
}
```

이것은 `색 이름표`를 만든 것이라고 생각하면 쉽습니다.

예를 들어 `--accent`는 강조 색입니다.
나중에 색을 바꾸고 싶으면 한 곳만 바꾸면 됩니다.

즉, 여러 군데 색을 직접 쓰는 것보다 관리가 쉬워집니다.

### 5-2. 왜 grid를 많이 썼는가

이번 뼈대에서는 카드 배치와 섹션 배치를 쉽게 하기 위해 `grid`를 많이 사용했습니다.

예를 들어:

- 상태 카드 4개 정렬
- 모임 카드 여러 개 정렬
- 왼쪽 설명 / 오른쪽 박스 배치

이런 구조는 `grid`가 이해하기 쉽고 수정도 편합니다.

### 5-3. 모바일 대응

아래 부분은 화면이 좁아질 때 한 줄씩 내려오게 만드는 코드입니다.

```css
@media (max-width: 960px) {
  .meeting-list {
    grid-template-columns: 1fr;
  }
}
```

뜻은 다음과 같습니다.

- 화면 너비가 960px 이하가 되면
- 모임 카드들을 3칸이 아니라
- 1칸씩 세로로 보여주겠다

그래서 휴대폰에서도 깨지지 않습니다.

## 6. script.js 설명

`script.js`는 화면을 실제로 움직이게 만드는 파일입니다.

여기서는 크게 4가지를 합니다.

1. 샘플 모임 데이터 저장
2. 화면에 카드 출력
3. 검색 기능 처리
4. 새 모임 추가 처리

### 6-1. 샘플 데이터

처음 부분에는 이런 데이터가 있습니다.

```javascript
const meetings = [
  {
    id: 1,
    title: "주말 풋살 같이 하기",
    category: "운동",
    location: "천안",
    date: "2026-03-20",
    capacity: 10,
    joined: 7,
    fee: "5,000원",
    mine: true,
  },
];
```

이것은 `가짜 모임 데이터`입니다.

아직 DB가 없기 때문에, 일단 JavaScript 안에 직접 데이터를 넣고 시작한 것입니다.

각 항목 뜻은 이렇습니다.

- `id`: 모임 구별 번호
- `title`: 모임 이름
- `category`: 카테고리
- `location`: 지역
- `date`: 날짜
- `capacity`: 모집 정원
- `joined`: 현재 참여 인원
- `fee`: 참가비
- `mine`: 내가 참여했는지 여부

### 6-2. 왜 render 함수를 만들었는가

`render`는 화면에 다시 그린다는 뜻입니다.

예를 들어:

- 처음 페이지가 열릴 때 카드 그리기
- 검색했을 때 조건에 맞는 카드만 다시 그리기
- 모임을 추가했을 때 새 카드까지 포함해서 다시 그리기

이런 일을 할 때 `render` 함수가 필요합니다.

대표 함수는 아래 3개입니다.

- `renderStatus()`
- `renderMeetings()`
- `renderMyMeetings()`

즉, 화면의 각 부분을 함수로 나눠서 다시 그릴 수 있게 만든 것입니다.

### 6-3. 검색 기능은 어떻게 동작하는가

검색 버튼을 누르면 이 코드가 실행됩니다.

```javascript
searchForm.addEventListener("submit", (event) => {
  event.preventDefault();
  filterMeetings();
});
```

아주 쉽게 풀면:

1. 검색 폼이 제출되는 순간을 잡는다
2. 기본 새로고침을 막는다
3. `filterMeetings()` 함수를 실행한다

`event.preventDefault()`가 중요한 이유는,
폼을 제출하면 원래 페이지가 새로고침되기 때문입니다.
우리는 지금 한 페이지 안에서 결과만 바꾸고 싶기 때문에 새로고침을 막았습니다.

### 6-4. filterMeetings()는 무엇을 하는가

이 함수는 입력값을 읽고, 조건에 맞는 데이터만 남깁니다.

흐름은 이렇습니다.

1. 키워드 입력값 읽기
2. 카테고리 선택값 읽기
3. 지역 선택값 읽기
4. `meetings` 배열을 하나씩 검사
5. 조건에 맞는 것만 새 배열로 만들기
6. 그 결과를 화면에 다시 출력

즉, `데이터를 고른 다음 다시 그린다`가 핵심입니다.

### 6-5. 모임 추가는 어떻게 되는가

폼에서 `모임 추가` 버튼을 누르면 `addMeeting()` 함수가 실행됩니다.

이 함수가 하는 일은 아래와 같습니다.

1. 폼의 기본 제출 동작 막기
2. 입력칸 값을 읽기
3. 새 객체 만들기
4. `meetings` 배열 맨 앞에 넣기
5. 폼 비우기
6. 화면 다시 그리기

코드의 핵심은 이 부분입니다.

```javascript
meetings.unshift(newMeeting);
```

`unshift`는 배열 맨 앞에 데이터를 넣는 뜻입니다.
그래서 새로 만든 모임이 목록 맨 위에 바로 보입니다.

### 6-6. 참여 신청 버튼은 왜 만들었는가

각 카드에는 `참여 신청` 버튼이 있습니다.

이 버튼은 지금 단계에서 아래 동작만 합니다.

- 참여 인원을 1 증가
- 내 모임 목록에도 들어가게 처리
- 화면 다시 그리기

즉, 진짜 서버 저장은 아니지만,
사용자 입장에서 어떤 흐름이 필요한지는 충분히 연습할 수 있습니다.

추가로 버튼 상태도 나눴습니다.

- 아직 참여하지 않았고 자리가 남아 있으면 `참여 신청`
- 이미 참여한 모임이면 `참여 완료`
- 정원이 가득 찼으면 `모집 마감`

이렇게 해 두면, 나중에 실제 서비스 동작을 생각하기가 더 쉬워집니다.

## 7. 지금 단계에서 꼭 이해해야 하는 핵심

웹 초보 입장에서 가장 중요한 흐름은 이것 하나입니다.

1. HTML로 입력칸과 보여줄 자리 만들기
2. JavaScript로 입력값 읽기
3. JavaScript로 데이터 바꾸기
4. 바뀐 데이터를 다시 화면에 출력하기

이 흐름이 이해되면,
나중에 DB를 붙여도 생각보다 구조가 크게 바뀌지 않습니다.

바뀌는 것은 주로 아래입니다.

- 지금은 JavaScript 배열에 직접 저장
- 나중에는 서버 DB에 저장

즉, 기본 사고방식은 거의 같습니다.

- 즉, 아직 자리가 남아 있다

1. 검색창 값 읽기
2. 상태 값 읽기
3. 배열을 한 개씩 검사
4. 조건에 맞는 것만 남기기
5. 정렬 순서대로 다시 배치
6. 화면에 다시 출력

즉, 이번에도 핵심은 똑같습니다.

`입력값 읽기 -> 데이터 고르기 -> 다시 출력`

## 14. 상세 보기 팝업은 왜 추가했는가

<dialog class="detail-dialog" id="detailDialog">

3. `showModal()`로 팝업 열기

즉, `선택한 카드의 정보를 dialog 안에 채워 넣는다`가 핵심입니다.

## 15. localStorage는 왜 중요한가

처음 만든 버전은 새로고침하면 추가한 모임이 전부 사라졌습니다.

그 이유는 데이터가 `JavaScript 메모리 안에만` 있었기 때문입니다.

그래서 이번에는 `localStorage`를 추가했습니다.

`localStorage`는 브라우저가 간단한 데이터를 저장해 두는 공간입니다.

쉽게 말하면:

- 페이지를 껐다 켜도
- 브라우저 안에 데이터가 남아 있는 작은 서랍

정도로 이해하면 됩니다.

### 15-1. 저장은 어떻게 하는가

모임 데이터가 바뀔 때 아래 함수를 실행합니다.

```javascript
function saveMeetings() {
  localStorage.setItem(STORAGE_KEY, JSON.stringify(meetings));
}
```

여기서 중요한 점은 2가지입니다.

- `localStorage`는 글자만 저장할 수 있다
- 그래서 `JSON.stringify()`로 배열을 문자열로 바꿔서 넣는다

### 15-2. 다시 불러오기는 어떻게 하는가

페이지가 처음 열릴 때 저장된 데이터가 있는지 먼저 확인합니다.

```javascript
const stored = localStorage.getItem(STORAGE_KEY);
```

만약 저장된 값이 있으면 `JSON.parse()`로 다시 배열로 바꿉니다.

즉, 흐름은 아래와 같습니다.

1. 저장할 때: 배열 -> 문자열
2. 불러올 때: 문자열 -> 배열

이 기능이 들어가면 사용자가 만든 모임이 새로고침 후에도 남아 있습니다.

이 방식이 좋은 이유는:

- 화면이 단순해지고

- 나중에 백엔드로 바꿀 때도 비슷한 구조를 유지하기 쉽기 때문입니다.

## 28. Figma 디자인을 어떻게 옮겼는가

오른쪽은 로그인 카드가 놓이는 영역입니다.

여기서 핵심은 카드가 `살짝 기울어져 보이는 느낌`입니다.

생각 흐름은 아래와 같습니다.

1. 바깥 상자를 살짝 기울인다
2. 안쪽 카드 본문은 반대로 기울인다
3. 그러면 전체적으로는 비스듬한 카드처럼 보인다

즉, 카드 전체를 복잡하게 그린 것이 아니라
`회전 효과`로 분위기를 만든 것입니다.

### 28-3. 입력 영역

입력칸은 아래 순서로 배치했습니다.

- Password

- Google 버튼

이 순서는 Figma 화면 흐름과 거의 같습니다.

## 29. login.js에서는 무엇을 했는가

이번 로그인 화면의 JavaScript는 크게 2가지만 합니다.

1. 비밀번호 보기 / 숨기기
2. 정적 데모 메시지 보여주기

### 29-1. 왜 실제 로그인 처리를 하지 않았는가

지금 단계는 아직 API를 제외하고 구현하는 단계이기 때문입니다.

즉, 로그인 디자인을 먼저 만들고
실제 로그인 통신은 나중에 붙이는 순서입니다.

이 순서가 좋은 이유는:

- 화면부터 먼저 완성할 수 있고
- 입력 흐름을 먼저 점검할 수 있고
- 나중에 백엔드를 붙일 자리가 분명해지기 때문입니다.

### 29-2. 비밀번호 보기는 어떻게 되는가

기본적으로 비밀번호 칸은 `type="password"`입니다.

버튼을 누르면:

- `password`를 `text`로 바꾸고
- 다시 누르면 `password`로 되돌립니다

즉, 입력칸 타입만 바꾸는 매우 기본적인 방식입니다.

이 패턴은 실제 서비스에서도 자주 씁니다.

이번 로그인 구현에서 기억해야 할 것은 이것입니다.

1. Figma 디자인은 먼저 `큰 덩어리`로 나눠서 본다

곧바로 로그인 기능까지 완성되는 것은 아닙니다.

- 전체 색감을 민트 계열에서 크림, 베이지, 브라운 계열로 바꿨다

### 왜 추가했는가

- 이전 메인은 깔끔했지만 일반적인 랜딩 페이지 느낌이 더 강했다
- 사용자가 참고한 UNESCO 사이트처럼 차분하고 전시형인 분위기를 일부 가져오고 싶었기 때문이다
- 하지만 현재 팀 수준과 프로젝트 구조상 3D, VR, 복잡한 스크롤 연출까지 그대로 따라가는 것은 과했기 때문이다
- 그래서 기능은 유지하고 시각 언어만 재해석하는 하이브리드 방식이 더 현실적이었다

### 어떻게 구현했는가

- `index.html`의 기능용 `id`는 그대로 두고 히어로 문구와 안내 블록 구조만 바꿨다
- `styles.css`에서 색상 변수를 전부 다시 정의해 전시형 팔레트로 교체했다
- 제목과 카드 머리글에는 세리프 계열 폰트를 사용해 전시 캡션 같은 분위기를 만들었다
- 모임 카드 상단의 큰 장식 영역은 없애고 대신 `ARCHIVE` 라벨과 얇은 프레임 선을 넣었다
- 버튼, 입력창, 팝업, 푸터까지 같은 톤으로 맞춰서 전체 화면이 한 스타일로 보이게 정리했다

### 이번 구현에서 중요하게 배운 점

- 참고 사이트를 그대로 복사하는 것이 아니라 서비스 목적에 맞게 해석해야 한다
- 디자인이 강한 사이트일수록 기능 구조를 먼저 지키고 시각만 가져오는 방식이 더 안전하다
- 지금 수준에서는 무거운 3D 효과보다 색, 여백, 타이포만 바꿔도 분위기가 크게 달라진다
