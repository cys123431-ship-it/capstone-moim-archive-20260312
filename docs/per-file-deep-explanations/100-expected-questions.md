# 예상 질문 100선

발표, 시연, 코드 설명 상황에서 나올 수 있는 질문을 100개로 정리한 문서다.
질문은 프로젝트 전체 구조, 메인 파일, 로그인 파일, 저장 방식, 배포, 발표 대응까지 고르게 포함했다.

## 1. 이 프로젝트는 전체적으로 어떤 사이트인가요?
- 답변: 모임을 찾고 참여하고 직접 개설하는 흐름을 보여 주는 정적 웹 프로토타입입니다. 현재는 서버 없이 HTML, CSS, JavaScript와 localStorage로 화면과 기본 동작을 먼저 구현한 상태입니다.

## 2. 이 프로젝트의 핵심 목적은 무엇인가요?
- 답변: 완성된 서비스 운영보다 웹 구조와 사용자 흐름을 먼저 구현하고 설명 가능한 상태로 만드는 것이 핵심 목적입니다. 그래서 검색, 참여, 추가, 수정, 삭제, 상세 팝업 같은 기본 흐름이 우선 들어가 있습니다.

## 3. 왜 HTML, CSS, JavaScript만 사용했나요?
- 답변: 현재 단계가 화면 구조와 상호작용 흐름을 먼저 검증하는 단계이기 때문입니다. 백엔드나 DB를 붙이기 전에도 프론트엔드 구조를 충분히 설명하고 시연할 수 있게 하려는 의도입니다.

## 4. 이 프로젝트에서 메인 페이지와 로그인 페이지를 나눈 이유는 무엇인가요?
- 답변: 역할이 다르기 때문입니다. 메인 페이지는 탐색과 관리가 중심이고 로그인 페이지는 입력 흐름과 인증 전 UI가 중심이라 구조와 스타일을 분리하는 편이 더 이해하기 쉽습니다.

## 5. 이 프로젝트에서 가장 중심이 되는 파일은 무엇인가요?
- 답변: `index.html`, `styles.css`, `script.js`입니다. 이 세 파일이 메인 화면의 구조, 디자인, 동작을 각각 맡습니다.

## 6. 현재 이 프로젝트는 실제 서비스인가요, 데모인가요?
- 답변: 지금은 데모에 가깝습니다. 실제 로그인 인증, 지도 API, 서버 DB 저장은 아직 연결하지 않았고, 프론트엔드 흐름을 먼저 구현한 상태입니다.

## 7. 메인 화면에서 사용자가 가장 먼저 보게 되는 것은 무엇인가요?
- 답변: 추천 배너와 빠른 탐색 타일입니다. 첫 화면에서 설명을 길게 읽기보다 바로 탐색을 시작하게 하려는 의도가 보입니다.

## 8. 검색 기능은 서버 검색인가요?
- 답변: 아닙니다. 현재는 `script.js` 안의 배열 데이터를 JavaScript가 직접 필터링하는 프론트엔드 검색입니다.

## 9. 정렬 기능은 어떤 기준으로 동작하나요?
- 답변: 가까운 날짜 순, 늦은 날짜 순, 남은 자리 많은 순, 참여 인원 많은 순으로 동작합니다. `sortMeetings()` 함수가 이 기준을 처리합니다.

## 10. 왜 추천 배너를 넣었나요?
- 답변: 첫 화면에서 대표 모임을 강조해서 사용자가 진입하자마자 선택지를 빠르게 보게 하려는 목적입니다. 단순 장식이 아니라 `script.js`가 현재 모임 데이터를 기준으로 배너를 갱신합니다.

## 11. 상단의 빠른 필터 버튼은 단순 디자인인가요?
- 답변: 아닙니다. `data-quick-filter` 값을 가진 실제 동작 버튼입니다. 클릭하면 검색 폼 상태와 추가 필터 규칙이 함께 바뀝니다.

## 12. 왜 카드형 목록으로 모임을 보여 주나요?
- 답변: 여러 모임을 짧게 훑기 쉽게 하기 위해서입니다. 제목, 위치, 카테고리, 인원, 참가비 같은 핵심 정보만 요약해서 한눈에 읽게 만듭니다.

## 13. 왜 상세 페이지 대신 팝업을 사용했나요?
- 답변: 카드에는 핵심 정보만 두고 자세한 설명은 필요할 때만 보여 주기 위해서입니다. 화면 이동 없이 바로 내용을 확인할 수 있다는 장점도 있습니다.

## 14. 왜 참여한 모임과 내가 만든 모임을 나눴나요?
- 답변: 참여 여부와 개설 권한은 다르기 때문입니다. `mine`은 참여 여부, `createdByMe`는 직접 개설 여부를 뜻하고 수정/삭제 권한은 후자와 연결됩니다.

## 15. 왜 추가와 수정을 같은 폼으로 처리했나요?
- 답변: 입력 구조가 거의 같기 때문에 같은 폼을 재사용하는 편이 단순하고 이해하기 쉽습니다. `editingMeetingId` 상태값으로 현재가 추가 모드인지 수정 모드인지 구분합니다.

## 16. 삭제 전에 확인창을 띄우는 이유는 무엇인가요?
- 답변: 데이터를 실수로 지우는 것을 막기 위해서입니다. 실제 서비스에서도 삭제는 보통 확인 단계를 한 번 더 둡니다.

## 17. localStorage는 왜 사용했나요?
- 답변: 아직 서버 DB가 없기 때문에 브라우저 안에라도 데이터를 저장해 새로고침 후에도 유지되게 하려는 목적입니다.

## 18. localStorage에 저장되는 데이터는 무엇인가요?
- 답변: 현재 모임 배열 전체가 JSON 문자열 형태로 저장됩니다. 즉 추가, 수정, 삭제, 참여 결과가 모두 이 저장값에 반영됩니다.

## 19. 페이지를 새로고침하면 어떤 일이 일어나나요?
- 답변: 브라우저가 HTML을 다시 읽고 CSS를 적용한 뒤 `script.js`가 실행됩니다. 그리고 `loadMeetings()`가 localStorage를 읽어 최신 데이터 상태를 복구합니다.

## 20. 모임 데이터는 어디에 정의되어 있나요?
- 답변: 초기 기본 데이터는 `script.js`의 `defaultMeetings` 배열에 들어 있습니다. 이후에는 localStorage 값이 우선됩니다.

## 21. 현재 사용자라는 개념은 어떻게 표현하나요?
- 답변: `CURRENT_USER` 상수를 사용해 가상의 현재 사용자 이름을 정해 두었습니다. 완전한 로그인 상태 관리 대신 데모용 기준값 역할을 합니다.

## 22. 왜 예전 이름을 바꾸는 보정 코드가 있나요?
- 답변: 초기 데이터나 저장 데이터의 표기 방식이 바뀌었을 때도 화면이 깨지지 않게 하려는 목적입니다. 예를 들어 예전 지역명과 주최자 이름을 현재 기준으로 바꾸는 매핑이 있습니다.

## 23. 왜 HTML과 CSS와 JS를 한 파일에 다 넣지 않았나요?
- 답변: 역할 분리가 되지 않으면 수정과 설명이 어려워집니다. 파일을 나누면 구조, 디자인, 동작을 각각 따로 이해하고 관리할 수 있습니다.

## 24. HTML의 가장 중요한 역할은 무엇인가요?
- 답변: 화면 뼈대를 만드는 것입니다. 검색 폼, 카드 목록, 팝업, 버튼, 링크 같은 기본 구조를 먼저 정의합니다.

## 25. CSS의 가장 중요한 역할은 무엇인가요?
- 답변: 같은 HTML 구조를 실제 서비스처럼 보이게 꾸미는 것입니다. 색, 여백, 정렬, 카드 모양, 반응형 배치를 모두 담당합니다.

## 26. JavaScript의 가장 중요한 역할은 무엇인가요?
- 답변: 사용자 행동을 읽고 데이터를 바꾼 뒤 다시 화면에 반영하는 것입니다. 현재 프로젝트에서는 검색, 추천, 참여, 추가, 수정, 삭제, 팝업이 여기에 해당합니다.

## 27. 왜 `defer`로 스크립트를 불러오나요?
- 답변: HTML이 먼저 파싱된 뒤 JavaScript가 실행되게 하려는 목적입니다. 그래야 `querySelector`가 찾을 DOM 요소가 이미 준비되어 있습니다.

## 28. 왜 `querySelector`와 `querySelectorAll`을 많이 쓰나요?
- 답변: HTML 안의 특정 요소를 JavaScript가 찾기 위해서입니다. 찾은 요소에 이벤트를 붙이거나 내용을 바꾸게 됩니다.

## 29. 왜 많은 곳에 `id`가 붙어 있나요?
- 답변: JavaScript가 정확히 어떤 요소를 조작할지 알아야 하기 때문입니다. 검색 폼, 목록, 팝업, 버튼 같은 핵심 요소는 `id`로 직접 찾는 경우가 많습니다.

## 30. 왜 class도 많이 쓰나요?
- 답변: CSS가 여러 요소에 공통 디자인을 적용하기 좋기 때문입니다. 예를 들어 `.button`, `.meeting-card`, `.meta-chip`처럼 같은 모양을 반복해서 쓸 수 있습니다.

## 31. 반응형 처리는 어디서 하나요?
- 답변: 메인 화면은 `styles.css`, 로그인 화면은 `login.css`의 미디어 쿼리에서 처리합니다. 화면 폭이 줄어들면 열 수, 패딩, 버튼 배치가 바뀝니다.

## 32. 현재 프로젝트에 백엔드가 없다면 한계는 무엇인가요?
- 답변: 다른 기기와 데이터 공유가 안 되고, 실제 회원 인증이나 서버 저장, 권한 검증 같은 기능을 할 수 없습니다. 지금은 브라우저 안에서만 상태가 유지됩니다.

## 33. 그렇다면 지금 단계에서 장점은 무엇인가요?
- 답변: 기능 흐름을 빠르게 시연하고 구조를 이해하기 쉽다는 점입니다. 서버 연결 전에도 사용자가 어떤 경험을 하게 될지 먼저 확인할 수 있습니다.

## 34. 검색 폼과 빠른 필터가 동시에 있는 이유는 무엇인가요?
- 답변: 정교한 조건 검색과 빠른 탐색을 둘 다 지원하려는 목적입니다. 세밀하게 고를 수도 있고, 추천/운동/공부처럼 한 번에 좁혀볼 수도 있습니다.

## 35. 추천 배너의 이전/다음 버튼은 어떤 의미가 있나요?
- 답변: 추천 목록 안에서 다른 대표 모임으로 이동하는 기능입니다. 단순 슬라이드가 아니라 현재 추천 데이터 배열을 기준으로 인덱스를 바꿉니다.

## 36. 추천 배너에서 '바로 참여' 버튼이 비활성화되는 경우는 언제인가요?
- 답변: 이미 참여한 모임이거나 모집이 마감된 모임일 때입니다. 현재 상태를 기준으로 버튼 가능 여부를 바꿉니다.

## 37. 검색 결과가 없을 때는 어떻게 처리하나요?
- 답변: 빈 상태 메시지를 카드 영역에 출력합니다. 사용자가 아무것도 보이지 않는 상황에서 헷갈리지 않도록 하기 위한 처리입니다.

## 38. 왜 숫자 상태 카드가 필요한가요?
- 답변: 전체 모임 수, 참여 인원, 내 모임 수, 남은 자리 수를 한눈에 보여 주기 위해서입니다. 데이터가 바뀌면 다시 계산해서 갱신합니다.

## 39. 폼을 초기화하는 버튼은 왜 필요한가요?
- 답변: 여러 조건을 시험하다가 빠르게 기본 상태로 돌아가기 위해서입니다. 검색 흐름을 시연할 때도 유용합니다.

## 40. 모임 설명을 카드에서 잘라서 보여 주는 이유는 무엇인가요?
- 답변: 카드 높이가 너무 길어지지 않게 하기 위해서입니다. 긴 설명은 상세 팝업에서 보여 주고, 카드에는 요약만 남깁니다.

## 41. 참여하기 버튼을 누르면 내부적으로 어떤 순서로 처리되나요?
- 답변: 대상 모임을 찾고, 정원이 남았는지와 이미 참여했는지 확인한 뒤, `joined`와 `mine` 값을 바꾸고 저장 후 다시 렌더링합니다.

## 42. 수정 버튼을 누르면 내부적으로 어떤 순서로 처리되나요?
- 답변: 선택한 모임 데이터를 찾아 폼에 채운 뒤 제목과 안내 문구를 수정 모드로 바꿉니다. 이후 제출하면 새로 추가하는 대신 해당 모임을 덮어씁니다.

## 43. 상세 팝업에서 수정/삭제 버튼이 어떤 때만 보이나요?
- 답변: 내가 만든 모임일 때만 보입니다. `createdByMe` 값을 기준으로 팝업 안 버튼 영역을 숨기거나 보여 줍니다.

## 44. 모임 날짜는 왜 문자열 형태로 저장하나요?
- 답변: HTML의 날짜 입력값과 바로 호환되고, 정렬할 때도 `new Date()`로 변환해 비교하기 쉽기 때문입니다.

## 45. 왜 참가비를 숫자가 아니라 문자열로 저장하나요?
- 답변: 현재는 `무료`, `각자 결제`, `5,000원`처럼 표현 방식이 다양하기 때문입니다. 데모 단계라 숫자 계산보다 표시가 더 중요합니다.

## 46. 에러 처리 코드는 어디에 있나요?
- 답변: 대표적으로 `loadMeetings()`와 `saveMeetings()`의 `try/catch`에서 localStorage 읽기/쓰기 실패를 대비합니다.

## 47. 현재 프로젝트에서 접근성을 위해 들어간 요소가 있나요?
- 답변: 예를 들어 `aria-label`, `aria-live`, `aria-hidden`, `lang="ko"` 같은 속성이 들어 있습니다. 완전한 접근성 구현은 아니지만 기본 배려는 보입니다.

## 48. 현재 프로젝트는 어떤 배포 방식을 고려하나요?
- 답변: GitHub Pages 같은 정적 호스팅입니다. `.nojekyll` 파일과 README의 배포 설명도 그 방향을 보여 줍니다.

## 49. 왜 README가 중요한가요?
- 답변: 처음 저장소를 보는 사람이 프로젝트 목적과 실행 방법을 빠르게 파악하게 해 주기 때문입니다. 코드보다 먼저 읽히는 안내서 역할을 합니다.

## 50. 왜 구현 설명 문서와 Q&A 문서가 따로 있나요?
- 답변: 구현 문서는 개발 흐름을 정리하고, Q&A 문서는 말하기 연습을 돕기 때문입니다. 같은 프로젝트라도 쓰임새가 다릅니다.

## 51. 발표에서 가장 먼저 설명해야 할 구조는 무엇인가요?

## 97. 왜 `assets/login-hero.png`를 다른 파일과 분리해 두었나요?
- 답변: 이 파일이 맡는 책임을 다른 파일과 섞지 않기 위해서입니다. 현재 역할은 '현재 화면에서는 미사용 상태다. 과거 로그인 시안 또는 후속 디자인 작업 자료로 보관된 자산으로 보인다.'라는 사용 시점과 연결되어 분리된 의미를 가집니다.

## 98. `assets/login-hero.png`는 주로 어떤 파일과 연결되나요?
- 답변: 대표 연결 대상은 다음과 같습니다. docs/web-implementation-guide.md에서 로그인 시안 자산으로 언급된다. 현재 index.html, login.html, styles.css, login.css에서는 이 이미지를 불러오지 않는다. 따라서 코드상 기준으로는 '보관 중인 자산'에 가깝다.

## 99. `assets/login-hero.png`가 없으면 가장 먼저 어떤 문제가 생기나요?
- 답변: 이 파일은 텍스트 코드가 아니라 자산 자체가 의미를 가진다.

## 100. 발표할 때 `assets/login-hero.png`는 한 문장으로 어떻게 소개하면 좋나요?
- 답변: 이 파일은 이미지 자산 파일입니다. 현재 배포 화면에서는 직접 쓰이지 않지만, 로그인 디자인 작업 과정의 흔적으로 남아 있습니다.
